<title>Spike: prove the Graphite PR-stacking sequence (gate)</title>
<description>
**Type:** Spike — time-boxed investigation that ends in a written decision. **Gates Tickets 1–4.**

## What

Drive the full Graphite (`gt`) + GitHub (`gh`) PR-stacking sequence by hand against one real repo with two stacked branches, and write down the exact commands that work in this environment.

## Why

`drain-cycle` has never touched Graphite or GitHub — no `gt`, `gh`, or PR code exists today. Before we wire the orchestrator to Graphite (Ticket 2), we need to know, not guess: whether `gt` runs from a *linked git worktree*, where a PR body comes from, and how a label gets applied. Building orchestration on unverified commands is the expensive way to discover they don't work here.

## How (mechanical)

Known-good mechanics from a prior manual run — reuse these, don't rediscover:

* `gt` needs `gt auth` (browser flow, one-time) **and** `gt init --trunk main` per repo. Auth failure aborts submit before any push.
* Adopt existing branches (don't recreate): `gt checkout <branch>` then `gt track --parent <parent>`. Parent = `main` for the stack base; the previous branch for each step up.
* `gt submit --no-interactive` creates **draft** PRs; add `--publish` for ready-for-review.
* `gt abort` refuses in non-interactive mode — use `git rebase --abort`.
* A branch forked off an older `main` shows "needs restack"; `gt restack` rebases it. A semantic conflict is a stop-the-line — abort and surface, never auto-merge.

Steps:

1. In a repo with `gt init --trunk main` and valid `gt auth`, create branch A off `main` and branch B off A, each with one commit.
2. From a **linked worktree**, run `gt track --parent main` (A) and `gt track --parent A` (B).
3. `gt submit --no-interactive --publish`; capture each PR number + URL (`gh pr view --json number,url` or `gt` output).
4. Create the label if absent (`gh label create review:high`), then apply it (`gh pr edit <n> --add-label review:high`).
5. Set a PR body — try the commit-message-derived body first, then `gh pr edit <n> --body-file <f>` as the fallback.

## Acceptance criteria

- [ ] A written decision (in this issue's comments or `docs/design-decisions.md`) records all four: (1) does `gt` work from a linked worktree, and which cwd the orchestrator will use; (2) the exact command sequence the orchestrator will run; (3) where the PR body comes from; (4) the per-repo auth/`gt init` preconditions.
- [ ] Two stacked PRs exist on GitHub in the correct parent order (B's base = A, A's base = `main`); one carries the `review:high` label; both bodies show What / Why / What to review.
- [ ] On `gt auth` or `gt init` failure, the spike stops and records the failure — it does not work around it.

**Precondition note:** `gt auth` is an interactive browser flow and cannot run under `--dangerously-skip-permissions`. Treat `gt auth` + `gt init --trunk main` as a one-time manual setup, not an automatable task.
